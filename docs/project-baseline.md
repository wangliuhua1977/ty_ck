# 项目基线说明（ty_ck）

## 1. 项目目标
构建一个基于 .NET 10 + WPF 的 Windows 桌面巡检系统，采用 Win11 原生风格与 Fluent 主题。

第一阶段目标：
- 平台同步
- 巡检
- 视频复核
- 企业微信通知

第一阶段不做：
- 地图主界面
- 工单/派单闭环
- 重点点位直播墙
- 复杂权限系统
- 重复通知抑制

## 2. 数据原则
- 平台是真源，本地库是“最新快照 + 人工主档 + 历史留痕”。
- 每次程序启动先同步平台最新数据。
- 每次巡检开始前，再同步本次巡检范围内最新数据。
- 平台字段刷新覆盖，本地人工字段长期保留。

## 3. 平台接入基线
- 公共请求参数固定 `version = "1.1"`。
- 请求头固定带 `apiVersion: 2.0`。
- 请求参数 `params` 走 XXTea。
- 响应 `data` 统一按 RSA 私钥解密。
- token 采用本地缓存与过期刷新。
- 点位详情主接口使用 `getDeviceInfoByDeviceCode`。

## 4. 点位组织方式
- 平台原始目录树只作为后台同步来源。
- 前台主结构使用用户自定义目录。
- 支持多个一级目录。
- 一个点位只能属于一个目录。

## 5. 人工维护字段
人工字段由管理员维护，并支持 CSV 导入导出：
- 点位别名
- 自定义目录
- 归属单位
- 维护负责人
- 维护人手机号
- 宽带接入号
- 宽带接入号状态
- 备注
- 是否启用巡检
- 停巡时段

## 6. 巡检判定规则
- 最终故障以“是否能稳定获取正常画面”为准。
- 在线状态、告警、设备详情只作辅助证据。
- 命中停巡窗口的点位，本轮直接跳过。

### 视频链路
- 主链路固定使用 RTSP。
- 单点位采用“两轮地址、四次播放尝试”的鲁棒策略。
- 完全无法正常播放时，也允许形成故障结果。

## 7. 截图与总览
- 巡检成功打开画面后抓取 2–3 张截图。
- 通知海报截图区域固定为三宫格。
- 总览页第一阶段只做截图墙。
- 重点点位直播墙不进入第一阶段开发。

## 8. 通知规则
- 系统只有一套通知机制，分为“巡检触发”和“人工触发”。
- 管理员可配置自动通知或人工审核后通知。
- 用户可以在任意点位上手工触发一次通知，无视当前状态。
- 企业微信通知样式固定为：
  1. 先 @ 维护人手机号
  2. 再发送单图海报
- 完全无法正常播放时，海报使用“无法正常播放”占位样式。

## 9. 页面范围
第一阶段页面：
- 首页
- 巡检任务
- 设备台账
- 总览（截图墙）
- 异常复核
- 通知记录
- 系统设置

### 异常复核页按钮
主按钮：
- 正常
- 故障
- 通知
- 下一条

辅助按钮：
- 重试
- 截图

## 10. 验证策略
采用分层验证策略：
- 界面轮：编译、启动、页面截图、关键点击验证。
- 普通功能轮：定向输入输出验证。
- 高风险轮：加密解密、视频流、通知必须做真实性测试。
- 阶段收口轮：跨模块联调。

## 11. Codex 开发方式
- 每轮单目标推进。
- 复杂任务先 plan 再 coding。
- 每轮使用新线程。
- 输出保持简洁：修改文件、完成项、未完成项、验证结果、剩余风险。
- 本地提交可以做，但由用户复审后手工 push。
