# AGENTS.md

## 项目定位
- 项目名称：ty_ck
- 技术栈固定：.NET 10、WPF、MVVM、Win11 原生风格、Fluent 主题。
- 第一阶段只做：平台同步、巡检、视频复核、企业微信通知。
- 第一阶段不做：地图主界面、工单闭环、重点点位直播墙、复杂权限系统、重复通知抑制。

## 开始任何工作前
1. 先阅读 `docs/project-baseline.md`。
2. 再阅读 `docs/ctyun-api/` 下的精简接口文档。
3. 如果本轮涉及企业微信通知，再阅读 `docs/integrations/企业微信Webhook说明.md`。
4. 复杂任务先给出简短计划，再开始编码。

## 长期固定规则
- 平台公共请求参数固定使用 `version = "1.1"`。
- 请求头固定带 `apiVersion: 2.0`。
- 请求参数 `params` 走 XXTea；响应 `data` 统一按 RSA 私钥解密。
- token 采用本地缓存与过期刷新，不允许每个接口前重复申请。
- 程序启动时同步平台最新数据；每次巡检开始前再同步本次巡检范围内最新数据。
- 点位详情主接口使用 `getDeviceInfoByDeviceCode`。
- 必须同步并保存平台经纬度，为后续地图接入预留。
- 前台不使用平台原始目录树作为主导航；界面使用用户自定义目录。
- 可维护多个一级目录；一个点位只能属于一个目录。
- 所有人工维护字段必须支持 CSV 导入导出。
- UI 和 webhook 消息统一使用简体中文 UTF-8。

## 巡检与判定规则
- 最终故障结论以“是否能稳定获取正常画面”为准，不以在线/离线为准。
- 在线状态、告警、设备详情只作为辅助证据。
- 视频主链路统一走 RTSP。
- 单点位巡检采用“两轮地址、四次播放尝试”的鲁棒策略。
- 完全无法正常播放也要形成结果，并允许通知。
- 命中停巡时间段的点位本轮直接跳过，不进入故障与通知。

## 通知规则
- 系统只有一套通知机制，区分“巡检触发”和“人工触发”。
- 管理员可配置自动通知或人工审核后通知。
- 企业微信通知样式固定为：先 @ 维护人手机号，再发送单图海报。
- 海报截图区域固定三宫格。
- 完全无法正常播放时，海报使用“无法正常播放”占位样式。
- 不要设计独立“派单系统”。人工“通知”只是复用同一套通知机制。

## 页面范围
第一阶段页面固定为：
- 首页
- 巡检任务
- 设备台账
- 总览（截图墙）
- 异常复核
- 通知记录
- 系统设置

## 复核页按钮
主按钮固定为：
- 正常
- 故障
- 通知
- 下一条

辅助按钮固定为：
- 重试
- 截图

## 验证策略（必须遵守）
采用分层验证策略：
- 界面轮：编译、启动、页面截图、关键点击验证。
- 普通功能轮：定向输入输出验证。
- 高风险轮（加密解密、视频流、通知）：必须做真实性测试。
- 阶段收口轮：做跨模块联调。

## Git 与工作方式
- 使用小步多轮开发；每轮只做一个清晰目标。
- 每一轮在新线程中开展。
- 可以在本地提交，但不要自动推送远程仓库。
- 由用户在复审后手工 push。
- 输出要克制：只给修改文件、完成项、未完成项、验证结果、剩余风险。

## 禁止事项
- 不要擅自扩大范围。
- 不要把平台原始目录树直接做成主 UI 结构。
- 不要把“在线=正常”或“离线=故障”写死为最终判断。
- 不要默认做地图页面。
- 不要默认做重点点位直播墙。
- 不要提交密钥、令牌、私钥或带密钥的本地配置。
